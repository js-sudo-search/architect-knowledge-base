Fundamental 1 – Chapter 4 – Functional Requirements

Coaching Record

Conversation 1 – Why do Functional Requirements exist?

Coach
Why do we need Functional Requirements? What information is still missing after defining the Business Objective?
Student
Functional Requirements define the capabilities required by the system. Non-Functional Requirements depend on them.
Diagnosis
The dependency was identified correctly, but the unique responsibility of Functional Requirements needed refinement.
Architect Refinement
Business Objectives answer Why. Functional Requirements answer What capabilities. Non-Functional Requirements answer How well.
Business Objective → Why?
Functional Requirement → What?
Non-Functional Requirement → How well?
Scale Estimation → How much?
Architecture → How?
Architecture Principle
Every layer contributes new information before architecture decisions are made.

Conversation 2 – Capability vs Implementation

Coach
How would you discover Functional Requirements for keeping customers informed about their orders?
Student
Initially proposed Email, SMS, live tracking and order history.
Diagnosis
The reasoning jumped to implementation before identifying capabilities.
Architect Refinement
First identify capabilities (track order, receive status updates, view order history). Only afterwards discuss implementation choices such as Email, SMS or Push Notifications.
Architecture Principle
Capabilities are defined before implementation choices.
Newsletter Opportunity
Don't Ask “Email or SMS?” Until You Know the Capability.

Conversation 3 – Functional Requirement vs UI Specification

Coach
Which is the Functional Requirement?
  1. The system shall allow users to upgrade their subscription.
  2. The Upgrade button should be blue.
Student
Initially classified the button specification as a Functional Requirement because it impacts user experience. After discussion, distinguished capability from UI design.
Diagnosis
Classification gap between capability and presentation.
Architect Refinement
Functional Requirements describe system capabilities. UI specifications describe presentation.
Architecture Principle
A capability remains valid even if the presentation changes.

Conversation 4 – Prioritizing Functional Requirements

Coach
How should Functional Requirements be prioritized?
Student
Prioritized Login, Reset Password and Dashboard before Export PDF, Multi-language and Dark Mode based on business continuity and business value.
Diagnosis
The prioritization was sound, but the criteria should be explained before the ordering.
Architect Refinement
Prioritize using: Business Continuity → Core Business Capability → Business Value → Dependencies → User Impact.
Architecture Principle
Architects justify priorities using explicit criteria rather than opinions.
Newsletter Opportunity
Why Not Every Functional Requirement Deserves Equal Priority.

Learning Capture

  • Shifted from feature thinking to capability thinking.
  • Separated capabilities from implementation choices.
  • Distinguished Functional Requirements from UI specifications.
  • Learned that Functional Requirements must be prioritized before deriving Non-Functional Requirements.
