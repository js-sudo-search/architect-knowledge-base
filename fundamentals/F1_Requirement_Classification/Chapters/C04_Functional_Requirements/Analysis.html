Fundamental 1 – Chapter 4 – Functional Requirements

Concept

Functional Requirements describe what capabilities the system must provide to achieve the Business Objective. They bridge the gap between business intent and technical design without prescribing implementation.

Key Insights

  • Business Objectives answer Why.
  • Functional Requirements answer What capabilities.
  • Non-Functional Requirements answer How well.
  • Architecture answers How.

Capability vs Implementation

Functional Requirement Implementation
The system shall notify users about order status changes. Email, SMS, Push Notification, WhatsApp, etc.
The system shall allow users to upgrade their subscription. Mobile App, Website, API, etc.

Functional Requirement vs UI Specification

Statement Classification
The system shall allow users to upgrade their subscription. Functional Requirement
The Upgrade button should be blue. UI / UX Specification
The password reset email should have a friendly tone. Content / Design Specification

Prioritization Framework

  1. Business Continuity
  2. Core Business Capability
  3. Business Value
  4. Dependencies
  5. User Impact

Common Mistakes

  • Confusing capabilities with implementation choices.
  • Confusing UI specifications with Functional Requirements.
  • Listing requirements without understanding the Business Objective.
  • Prioritizing based on opinion instead of explicit criteria.

Interview Guidance

When gathering Functional Requirements, avoid asking "Should we use Email or SMS?" Instead ask:

  • What capability should the system provide?
  • Who needs this capability?
  • Why is it valuable to the business?
  • Is this capability essential for the first release?
