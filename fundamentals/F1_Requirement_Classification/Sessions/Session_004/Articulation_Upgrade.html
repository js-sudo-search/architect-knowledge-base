Session 004 – Articulation Upgrade (Detailed)

Problem 1

1. Interview Problem
Why does it matter architecturally whether the URL Shortener is global or regional?
2. Your Original Answer
If the geographical locations are distributed, it impacts the architectural decision because then we have to think about the number of servers that can support the availability, the redundancy, durability. So these are the things that will impact the architectural designs.
3. Gaps
• Correct intuition. • Jumped directly to server count. • Missing causal chain: Geography → Longer distance → Higher latency → Regional deployment → Traffic routing. • Explained outcomes before explaining why.
4. Refined Architect Answer
If the service is globally distributed, the first architectural consequence is increased network latency because requests travel longer distances. To reduce latency, services are deployed closer to users and traffic is routed to the nearest healthy region. Multi-region deployment also improves resilience through regional failover.

Problem 2

1. Interview Problem
The service has 500 million monthly active users. Would you start designing immediately or perform scale estimation first? Why?
2. Your Original Answer
This is a global URL shortener with 500 million active users. Some users are nearer to servers while others are farther away, so latency changes. Users become active at different times across regions. If one region fails, traffic should be routed to nearby servers. Therefore resilience and high availability become important.
3. Gaps
• Stayed in NFR discussion. • Didn't answer why estimation precedes architecture. • Missing idea: estimation provides evidence such as QPS, storage and throughput before design.
4. Refined Architect Answer
Before proposing an architecture, I would estimate workload characteristics such as read/write QPS, storage growth and throughput. Those estimates determine whether a simple architecture is sufficient or whether distributed components are required. Without them, architecture would be based on assumptions instead of evidence.

Problem 3

1. Interview Problem
If you could estimate only four things before starting the architecture, which four would you choose, and why?
2. Your Original Answer
The four most influential metrics that would help me do a scale estimation... First, per geographical location, what is the number of active users? What is the peak traffic? What is the size of each request? That will give me an idea about storage. The first one will give me an idea about query per second. The fourth would be network limitations. I need bandwidth so that I can understand throughput. Based on throughput I can decide the number of backend servers. The fourth would be the sensitivity of the functionality. Do we need this functionality to be highly available?
3. Gaps
• Mixed inputs (active users, request size) with derived metrics (QPS, storage, throughput). • Availability is an NFR, not a scale metric. • Did not state which architectural decision each metric enables.
4. Refined Architect Answer
I would estimate: 1. Read/Write QPS — to determine horizontal scaling. 2. Storage growth — for capacity planning. 3. Throughput/Bandwidth requirements — for network sizing. 4. Peak versus Average Load — for headroom and auto-scaling. Each metric directly justifies an architectural decision.

Interview Principle

Requirement → Constraint → Evidence → Architectural Decision → Technology.
