Session 002 – Coaching Transcript

Fundamental: F1 – Requirement Classification

Chapter: C04 – Functional Requirements

Dialogue 1

Coach
Business Objective: "Improve customer experience by keeping customers informed throughout the order lifecycle."

Why do Functional Requirements exist? What information is still missing before architecture?
Student
Functional Requirements help understand the user's needs. They define the capability required by the system and Non-Functional Requirements depend on them.
Coach's Diagnosis
Correct dependency identified, but the unique purpose of Functional Requirements was still unclear.
Architect Refinement
Business Objectives answer Why.
Functional Requirements answer What capabilities.
Non-Functional Requirements answer How well.

Dialogue 2

Coach
How would you discover the Functional Requirements?
Student
Should users receive Email or SMS updates?
Do they want live tracking?
Do they want order history?
Coach's Diagnosis
The reasoning jumped directly to implementation choices. Email and SMS are technologies, not Functional Requirements.
Architect Refinement
First discover capabilities:
  • Should users know the current order status?
  • Should users be notified when the status changes?
  • Should users view order history?
Implementation comes afterwards.

Dialogue 3

Coach
Business Objective: Improve subscription management. List Functional Requirements.
Student
Allow users to cancel subscriptions.
Show subscription history.
Allow users to subscribe again.
Allow users to upgrade subscriptions.
Coach's Diagnosis
Good capability thinking. One observation was distinguishing capabilities from business rules.
Architect Refinement
Phrase Functional Requirements from the system's perspective: "The system shall allow users to upgrade subscriptions."

Dialogue 4

Coach
Which statement is a Functional Requirement?
  1. The system shall allow users to upgrade subscriptions.
  2. The Upgrade button should be blue.
Student
Initially identified the button specification as a Functional Requirement because it affects user experience.
Coach's Diagnosis
Classification gap. UI specifications are not Functional Requirements.
Architect Refinement
Capabilities survive UI redesigns; UI specifications do not.

Dialogue 5

Coach
How would you prioritize Functional Requirements for a two-month release?
Student
Prioritized Login, Reset Password and Dashboard before Export PDF, Multi-language and Dark Mode based on business continuity and business value.
Coach's Diagnosis
Good prioritization, but explain the prioritization criteria before presenting the ordering.
Architect Refinement
Use explicit criteria:
  • Business Continuity
  • Core Business Capability
  • Business Value
  • Dependencies
  • User Impact

Session Closing

The major evolution during this session was the transition from feature thinking to capability thinking. By the end of the chapter, Functional Requirements were consistently treated as system capabilities derived from Business Objectives rather than implementations or UI specifications.
