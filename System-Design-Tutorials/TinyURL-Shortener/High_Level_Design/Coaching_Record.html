TinyURL – High Level Design

Coaching Record

Conversation 1 – Where should a High-Level Design begin?

Coach
You have completed requirement gathering and capacity estimation. Draw the first version of TinyURL. Do not think about Redis, Kafka, Load Balancers or database sharding. Build the smallest system that works.
Student
Client → URL Receiver → Generate Short URL → Store URL Mapping → Database → Return Short URL.

For redirection:
Client → URL Receiver → Lookup Mapping → Database → Return Long URL.
Diagnosis
Excellent architectural discipline. The design started with the minimum viable architecture instead of introducing scalability components prematurely.
Architect Refinement
A High-Level Design should begin with the simplest architecture capable of satisfying the current requirements. Complexity should emerge only when the current design becomes insufficient.
Architecture Principle
Start simple. Evolve only when necessary.
Newsletter Opportunity
Most engineers start with Redis. Architects start with a working system.

Conversation 2 – When should multiple application servers be introduced?

Coach
Suppose the number of users grows significantly. When would you introduce multiple URL Receiver instances?
Student
That depends on how much load a single URL Receiver can handle. Capacity estimation provides the expected QPS, and profiling the service tells us whether one instance is sufficient.
Diagnosis
Strong architectural reasoning. Capacity estimation was correctly used as an input to architectural decisions instead of treating it as an isolated exercise.
Architect Refinement
Capacity estimation answers "How much load should the system expect?" Profiling answers "Can the current architecture sustain that load?" Architecture evolves only when both answers indicate the need.
Architecture Principle
Estimate first. Measure second. Scale third.

Conversation 3 – Why introduce a Load Balancer?

Coach
Suppose CPU profiling shows sustained saturation and you decide to add five URL Receiver instances. What new architectural problem appears?
Student
Once multiple URL Receiver instances exist, requests must be directed to one of them. The system now needs a mechanism to make that routing decision.
Diagnosis
The reasoning focused on the underlying problem instead of immediately naming a component. This demonstrates architectural thinking rather than pattern memorization.
Architect Refinement
A Load Balancer is not introduced because multiple servers exist. It is introduced because horizontal scaling creates the request routing problem.
Architecture Principle
Components exist to solve problems—not to complete architecture diagrams.
Newsletter Opportunity
The Load Balancer Is Not the Design. The Routing Problem Is.

Conversation 4 – Should the database also have a Load Balancer?

Coach
If application servers require a Load Balancer, should we also place one in front of the database?
Student
A single database may eventually become a bottleneck because it must handle all read and write requests. In that situation, distributing requests across multiple databases could help.
Diagnosis
The observation is technically correct, but the architectural sequence was premature. The database had not yet been identified as the bottleneck.
Architect Refinement
Do not introduce new components because they might become useful later. First demonstrate that the current design is insufficient using measurable evidence such as CPU utilization, query latency, connection count, or I/O saturation.
Architecture Principle
Evidence should drive architecture—not anticipation.

Learning Capture

  • High-Level Design begins with the simplest working architecture.
  • Capacity estimation naturally feeds into architecture decisions.
  • Horizontal scaling introduces the request routing problem.
  • A Load Balancer is a consequence of horizontal scaling, not a starting component.
  • Avoid solving future problems before observing evidence that they exist.
  • Architectural evolution should follow the sequence: Problem → Evidence → Decision.
